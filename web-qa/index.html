QA Process — WordPress Site

QA process covering functionality, performance, SEO & WCAG 2.1 AA
Prepared:
Environment: Staging

Quick Start — Environment & Smoke Test

  1. Confirm environments: staging URL (not production), credentials for Admin, Editor, Subscriber/test user.
  2. Browser / device matrix: note devices and browsers to test (see Responsive section).
  3. Smoke test (15–30 min): load main pages (home, blog list, single post, contact, about). Check critical flows: login, contact form, search, newsletter signup, plugin-specific features. If a critical failure appears, mark as Blocker and stop deeper QA.

Key Testing Areas

  • Functional / Feature testing
  • Responsive / Cross-browser
  • Performance & Core Web Vitals
  • SEO & structured data
  • Accessibility — WCAG 2.1 AA
  • Security & privacy basics
  • Content / copy / media validation
  • Plugin & theme integration / regression
  • Usability / visual QA / style
  • Release readiness & sign-off

Detailed Tests, Tools & Methods

1. Functional / Feature testing

What to check

  • All pages load; nav/menu/footer links; 404 handling.
  • Forms (contact, comment, signup): client & server validation, success messages, email receipts.
  • Authentication: login, logout, password reset, role-based access (Admin/Editor/Subscriber).
  • Plugin features: test both custom plugins end-to-end and third-party plugin behavior.
  • CRUD for posts/pages; search, pagination, category/tag pages; RSS & sitemap.xml.

Tools & methods

  • Manual test matrix (spreadsheet) with test cases and pass/fail.
  • Browser DevTools network tab & console for Ajax/JS errors.
  • Automated E2E: Playwright or Cypress for critical flows (login, contact form, publish post).
  • WP-CLI for DB/option inspection if needed.

2. Responsive & Cross-browser

What to check

  • Layout at key viewports: mobile, tablet, desktop. Tap targets, font sizes, image scaling, hamburger menu behavior.
  • Breakpoint behavior for all theme components.

Tools & methods

  • DevTools device toolbar for quick checks; BrowserStack or Sauce Labs for real-device cross-browser testing.
  • Minimum test list: Chrome, Firefox, Safari (macOS & iOS), Edge, Android Chrome (real devices where possible).

3. Performance & Core Web Vitals

What to check

  • LCP, FID/INP, CLS; page weight; number of requests; TTFB.

Tools & methods

  • Lighthouse (Chrome DevTools) per page (home, blog list, single post, contact).
  • WebPageTest for realistic throttling and filmstrip.
  • PageSpeed Insights for field + lab data.
  • Server-side: Query Monitor, New Relic for slow queries and bottlenecks.

4. SEO

What to check

  • Unique title tags & meta descriptions, correct heading structure (H1 per page), meta robots & canonical tags, sitemap.xml, robots.txt.
  • Article schema for blog posts, Organization schema for site, image alt text, social tags (OpenGraph/Twitter).

Tools & methods

  • Screaming Frog to crawl for missing tags / duplicates / broken links.
  • Google Search Console property & sitemap submission.
  • Rich Results Test for structured data validation.

5. Accessibility — WCAG 2.1 AA

Priority checks

  • Color contrast ratios (>= 4.5:1 for normal text), keyboard navigation, correct ARIA roles & landmarks, form labels & error messaging, skip-to-content link, focus management on modals, no reliance on color alone.
  • Captions/transcripts for multimedia if present.

Tools & methods

  • Axe DevTools browser extension for automated checks.
  • WAVE and Accessibility Insights to surface common issues.
  • Manual testing with NVDA (Windows) and VoiceOver (macOS/iOS); test zoom (200%) and increased text size.

6. Security & Privacy (basic)

  • WP_DEBUG off in production, remove unused plugins/themes, strong salts in wp-config.php, correct file permissions.
  • SSL/TLS present, HSTS header, disable XML-RPC if unused, review REST API exposures.

Tools & methods

  • WPScan for known vulnerabilities (run with permission on staging), header checks for security headers.
  • Verify GDPR/privacy considerations for form data and cookie consent.

7. Content / Media QA

  • Spellcheck and proofread all 20 blog posts, verify featured images and alt text, validate internal/external links, author attribution, published dates, categories/tags.
  • Tools: browser spellchecker, Screaming Frog for broken links, spreadsheet checklist per post.

8. Plugin & Theme Integration

  • Confirm custom theme and plugins don’t conflict with third-party plugins; test plugin settings persist and front-end effects; inspect console & PHP logs for errors.
  • Tools: Query Monitor, WP_DEBUG_LOG on staging, PHPUnit (if unit tests exist for custom plugins).

Test Plan Prioritization & Matrix

Priority definitions

  • P0 (Blocker): site down, critical flows fail, major accessibility keyboard failure, severe performance regression.
  • P1 (High): SEO-critical missing tags, major mobile visual break, JS errors blocking features.
  • P2 (Medium): minor accessibility contrast issues, non-critical missing alt text, slow queries.
  • P3 (Low): typos, small analytics misconfig, style polish.

Automation & CI Suggestions

  • Run Lighthouse CI in GitHub Actions on PRs for main templates/pages.
  • Use Playwright/Cypress for smoke E2E tests on deploy.
  • Static checks: PHPCS (WP Coding Standards), ESLint for JS.
  • Run Axe accessibility checks in CI to catch regressions early.

Defect Reporting & Workflow

  • Use issue tracker (Jira/GitHub Issues/Trello). Triage daily: devs fix → QA verify → close.
  • Every bug: reproducible steps, environment, severity, attachments (screenshots / logs).
  • Blockers require immediate hotfix on staging, then retest before production deploy.

Release Readiness & Sign-off Criteria

  • All P0 issues resolved and verified.
  • ≥90% of P1 resolved OR documented client acceptance of remaining P1s.
  • Accessibility: critical WCAG 2.1 AA violations fixed; remaining items documented with remediation plan.
  • Performance baseline recorded; recommended thresholds met (client-defined).
  • SEO: sitemap & robots present; no duplicate title/H1 issues; article schema implemented.
  • Security checklist completed (SSL, perms, unused plugins removed).
  • Final smoke test on production post-deploy & stakeholder sign-off.
